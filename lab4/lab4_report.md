## Описание компонентов MVP

### User

Пользователь приложения. Он загружает изображение товара или вводит текстовое описание.

### Frontend

Frontend — это web-интерфейс, через который пользователь взаимодействует с приложением. Он отвечает за загрузку изображений, отправку запросов и отображение результата генерации.

### Backend API Cloud Run

Backend API размещается в Cloud Run. Он принимает запросы от frontend, обрабатывает данные, обращается к AI API, сохраняет файлы и результаты генерации.

Cloud Run выбран потому, что он позволяет запускать контейнеризированное приложение без ручного управления серверами. Оплата Cloud Run зависит от фактически использованных ресурсов, а ресурсы округляются с точностью до 100 миллисекунд. Это удобно для MVP, где нагрузка нерегулярная. ([Google Cloud][2])

### AI API

AI API используется для генерации описаний товаров. В качестве AI API можно использовать Gemini API или OpenAI API. Для экономической модели в данной работе используется модель с оплатой по токенам, так как AI API обычно тарифицируются по количеству входных и выходных токенов. Например, в документации Gemini API указано, что биллинг основан на количестве input tokens, output tokens, cached tokens и длительности хранения cached tokens. ([Google AI for Developers][3])

### Cloud SQL

Cloud SQL используется для хранения структурированных данных:

* пользователей;
* истории генераций;
* статусов запросов;
* результатов генерации.

Cloud SQL подходит для MVP, потому что приложение работает со структурированными данными, а в дальнейшем эту базу можно масштабировать без изменения архитектуры. Стоимость Cloud SQL зависит от выбранной редакции, количества CPU, памяти и объёма хранилища. ([Google Cloud][4])

### Cloud Storage

Cloud Storage используется для хранения изображений товаров. Это объектное хранилище, которое хорошо подходит для файлов, изображений и других неструктурированных данных. Стоимость Cloud Storage зависит от класса хранения, операций, объёма данных и передачи данных. ([Google Cloud][5])

### Secret Manager

Secret Manager используется для хранения API-ключей и секретных данных. Это безопаснее, чем хранить ключи в исходном коде приложения.

### Cloud Logging

Cloud Logging используется для сбора логов backend-приложения. Это помогает находить ошибки и анализировать работу приложения.

Для 1 500 генераций в месяц:

`1 500 × $0.00172 = $2.58`

## Обоснование выбора решения для MVP

На этапе MVP самым важным является быстро проверить идею и не тратить ресурсы на сложную инфраструктуру.

Поэтому были выбраны:

* **Cloud Run**, а не Kubernetes, потому что Cloud Run проще и дешевле в поддержке;
* **Cloud SQL**, потому что приложение работает со структурированными данными;
* **Cloud Storage**, потому что изображения лучше хранить в объектном хранилище, а не в базе данных;
* **Secret Manager**, потому что API-ключи нельзя хранить в коде;
* **Cloud Logging**, потому что даже на MVP нужно иметь возможность отслеживать ошибки.

Самым дешёвым вариантом мог бы быть один сервер или локальная база данных, но такое решение сложнее масштабировать. Поэтому для MVP выбрана инфраструктура, которая не является избыточной, но при этом может развиваться дальше.

## Описание изменений по сравнению с MVP

Главное отличие Partner Testing от MVP — появление очереди задач и отдельного AI Worker.

В MVP backend напрямую обращался к AI API. На этапе тестирования партнёрами такой подход становится менее надёжным, потому что AI-запросы могут выполняться долго. Если backend будет ждать ответ AI API, пользователь может столкнуться с долгим ожиданием, а backend может быть перегружен.

Поэтому используется асинхронная обработка:

1. Пользователь отправляет запрос.
2. Backend сохраняет данные и ставит задачу в очередь.
3. AI Worker забирает задачу из очереди.
4. Worker обращается к AI API.
5. Результат сохраняется в Cloud SQL.
6. Пользователь получает готовый результат после завершения обработки.

---

## Обоснование дополнительных ресурсов

### Cloud Tasks / Pub/Sub

Очередь используется для асинхронной обработки AI-задач. Это позволяет backend быстро принять запрос и не ждать длительный ответ от AI API.

Pub/Sub тарифицируется по объёму опубликованных, доставленных и сохранённых данных. Также у Pub/Sub есть бесплатный объём: первые 10 GB данных в месяц не тарифицируются, а дальнейшая стоимость зависит от объёма. ([Google Cloud][7])

### AI Worker Cloud Run

AI Worker — отдельный сервис, который занимается только обработкой AI-запросов. Он забирает задачи из очереди, отправляет запросы в AI API и сохраняет результаты.

Такое разделение делает систему стабильнее: backend отвечает за пользовательские запросы, а worker — за долгие AI-операции.

### Redis Cache

Redis Cache используется для хранения часто запрашиваемых данных и статусов задач. Например, frontend может часто проверять статус генерации. Чтобы каждый раз не обращаться к базе данных, статус можно хранить в Redis.

Memorystore for Redis тарифицируется по выделенной памяти и времени работы. Например, в документации Google приведён расчёт для Basic Tier 8 GiB, где стоимость составляет около $160.60 в месяц. Поэтому на этапе Partner Testing Redis можно считать опциональным ресурсом и включать его только при необходимости. ([Google Cloud][8])

### Cloud Monitoring

Cloud Monitoring добавляется для отслеживания метрик приложения:

* количество запросов;
* ошибки backend;
* время ответа;
* состояние worker;
* количество задач в очереди.

## Обоснование выбора решения для Partner Testing

Для Partner Testing важно найти баланс между стоимостью и стабильностью.

Самое дешёвое решение — оставить архитектуру MVP и напрямую обращаться к AI API из backend. Но это может привести к проблемам при росте нагрузки: backend будет дольше обрабатывать запросы, а пользователи будут ждать ответа.

Поэтому более оптимальным решением является добавление очереди и worker-сервиса. Это немного увеличивает стоимость инфраструктуры, но делает приложение более устойчивым и готовым к дальнейшему росту.

## Описание компонентов Production

### Cloud Load Balancer

Cloud Load Balancer используется для распределения входящего трафика и обеспечения стабильной работы приложения при большом количестве пользователей.

Для load balancing Google Cloud тарифицирует forwarding rules. Например, в официальной документации указано, что до 5 forwarding rules могут стоить $0.025 в час. ([Google Cloud][9])

### Cloud Armor

Cloud Armor используется для защиты приложения от нежелательного и потенциально вредоносного трафика. Это особенно важно на production-этапе, когда приложение доступно реальным пользователям.

Cloud Armor имеет отдельную модель тарификации, которая зависит от выбранного режима, правил, запросов и объёма обработанных данных. ([Google Cloud][10])

### Cloud CDN

Cloud CDN используется для ускорения загрузки frontend и статических файлов. Это снижает задержки для пользователей и уменьшает нагрузку на origin-сервисы.

Стоимость Cloud CDN зависит от объёма передаваемых данных, cache fill и количества запросов. ([Google Cloud][11])

### Cloud SQL PostgreSQL HA

В production используется Cloud SQL PostgreSQL в режиме High Availability. Это дороже обычной базы, но повышает надёжность приложения.

HA-режим нужен, потому что база данных становится критически важным компонентом: в ней хранятся пользователи, история генераций, статусы задач и результаты.

### Automated Backups

Automated Backups используются для резервного копирования базы данных. Это необходимо для восстановления данных после сбоев или ошибок.

### Memorystore Redis

Redis используется для кэширования:

* статусов задач;
* часто используемых данных;
* пользовательских сессий;
* промежуточных результатов.

В production Redis становится более важным, потому что он снижает нагрузку на Cloud SQL и ускоряет ответы приложения.

# Почему не выбрано самое дешёвое решение

На первом этапе можно было бы выбрать максимально дешёвую инфраструктуру: один сервер, локальную базу данных или хранение данных в файлах. Однако такое решение плохо масштабируется и потребует серьёзной переработки при переходе к партнёрскому тестированию или production.

В данной работе выбрано решение, которое остаётся относительно недорогим на этапе MVP, но при этом может развиваться дальше без полного изменения архитектуры.

Основные причины такого выбора:

* Cloud Run можно использовать на всех трёх этапах.
* Cloud SQL можно масштабировать от небольшой базы до HA-конфигурации.
* Cloud Storage подходит как для MVP, так и для production.
* Очередь и worker можно добавить позже, не переписывая всю систему.
* Secret Manager и Logging нужны уже на раннем этапе, потому что помогают соблюдать безопасность и находить ошибки.
* Redis не обязателен на MVP, но полезен на этапах Partner Testing и Production.
* Load Balancer, Cloud Armor и CDN не нужны на MVP, но важны в production.

Таким образом, инфраструктура развивается постепенно: от простой MVP-схемы к более устойчивой и масштабируемой production-архитектуре.

# Обоснование выбора ресурсов

## Cloud Run

Cloud Run выбран для backend API и AI Worker, потому что он поддерживает serverless-подход, автоматически масштабируется и не требует ручного управления серверами. Это особенно удобно для AI-приложения, где нагрузка может быть неравномерной.

## Cloud SQL PostgreSQL

Cloud SQL PostgreSQL выбран для хранения структурированных данных. В приложении есть пользователи, задачи генерации, история запросов и результаты. Для таких данных реляционная база подходит лучше, чем объектное хранилище.

## Cloud Storage

Cloud Storage используется для хранения изображений товаров. Изображения являются объектными данными, поэтому их нецелесообразно хранить в базе данных.

## Pub/Sub / Cloud Tasks

Очередь нужна для асинхронной обработки AI-запросов. Это повышает устойчивость приложения при росте нагрузки и позволяет backend не ждать долгий ответ от AI API.

## AI Worker Cloud Run

AI Worker выделен в отдельный сервис, чтобы отделить пользовательские запросы от долгих операций генерации текста. Это упрощает масштабирование и повышает стабильность.

## Secret Manager

Secret Manager используется для безопасного хранения API-ключей и секретов. Это снижает риск утечки конфиденциальных данных.

## Cloud Monitoring и Cloud Logging

Monitoring и Logging необходимы для анализа состояния приложения, поиска ошибок и контроля производительности.

## Memorystore Redis

Redis используется для кэширования статусов задач и часто используемых данных. Это снижает нагрузку на Cloud SQL и ускоряет работу приложения.

## Cloud Load Balancer

Load Balancer используется в production для распределения входящего трафика и повышения стабильности приложения.

## Cloud Armor

Cloud Armor добавлен в production для защиты от нежелательного трафика и атак.

## Cloud CDN

Cloud CDN ускоряет загрузку frontend и статических файлов, что улучшает пользовательский опыт.

## Automated Backups

Automated Backups нужны для восстановления данных в случае сбоя, ошибки или потери данных.
