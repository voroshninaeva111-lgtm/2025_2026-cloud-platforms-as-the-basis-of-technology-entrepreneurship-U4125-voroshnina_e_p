## Цель работы

Ознакомиться с работой сервиса Google Cloud Run, изучить процесс развертывания контейнерного приложения, протестировать работу сервиса, проанализировать логи и метрики, а также проверить работу ревизий и переключения трафика между ними.

## Ход работы

### 1. Создание сервиса Cloud Run

Для выполнения лабораторной работы был использован сервис **Cloud Run** в проекте Google Cloud `cloud-platforms-as-the-basis`.

В разделе **Cloud Run → Services** был создан новый сервис с именем:

`evoroshnina-cloudrun-lab2`

Для развертывания был использован стандартный контейнерный образ Hello:

`us-docker.pkg.dev/cloudrun/container/hello`

При создании сервиса были указаны следующие параметры:

`Region: europe-west1`

`Authentication: Allow public access`

`Billing: Request-based`

`Minimum number of instances: 0`

`Maximum number of instances: 1`

`Container port: 8080`

`Memory: 512 MiB`

`CPU: 1`

Таким образом, сервис был создан с минимальным количеством ресурсов.

### 2. Проверка созданной ревизии

После создания сервиса Cloud Run была автоматически создана первая ревизия:

`evoroshnina-cloudrun-lab2-00001-8gb`

В настройках ревизии было видно, что контейнер использует порт:

`8080`

Также были указаны параметры контейнера:

`CPU limit: 1`

`Memory limit: 512 MiB`

После завершения деплоя сервис получил публичный URL:

`https://evoroshnina-cloudrun-lab2-307056602443.europe-west1.run.app`


### 3. Тестирование сервиса

После перехода по URL Cloud Run была открыта стандартная страница Hello-сервиса.

На странице отобразилось сообщение:

`It's running!`

Это означает, что контейнер был успешно развернут в Cloud Run и корректно обработал HTTP-запрос.

Также на странице была указана созданная ревизия:

`evoroshnina-cloudrun-lab2-00001-8gb`

После тестирования сервиса был открыт раздел **Observability → Logs**.

В логах были отображены события, связанные с созданием сервиса, запуском контейнера и обработкой HTTP-запросов.

В частности, в логах были видны следующие события:

`CreateService`

`Starting new instance`

`The container started successfully and is listening for HTTP requests on port 8080`

`GET 200`

Записи `GET 200` подтверждают, что сервис успешно обработал запросы пользователя и вернул корректный HTTP-ответ.

### 5. Анализ метрик

Далее был открыт раздел **Observability → Metrics**.

В разделе метрик были отображены графики, связанные с работой Cloud Run сервиса:

Метрики позволяют отслеживать количество запросов, задержку ответа, состояние контейнера и параметры масштабирования сервиса.

### 6. Изменение порта контейнера

После анализа логов и метрик была создана новая ревизия сервиса.

Для этого была использована функция:

`Edit & deploy new revision`

В настройках контейнера порт был изменен с:

`8080`

на:

`8090`

После сохранения изменений была создана новая ревизия:

`evoroshnina-cloudrun-lab2-00002-bmr`

В настройках новой ревизии было видно, что контейнер использует порт:

`8090`


### 7. Проверка сервиса после изменения порта

После изменения порта был повторно открыт URL сервиса Cloud Run.

Ожидалось, что сервис может перестать работать, так как изначально контейнер использовал порт `8080`.

Однако сервис продолжил успешно работать и снова отобразил страницу:

`It's running!`

На странице была указана новая ревизия:

`evoroshnina-cloudrun-lab2-00002-bmr`

Это связано с тем, что стандартный Hello-контейнер Cloud Run использует переменную окружения `PORT`. При изменении порта Cloud Run передал контейнеру новое значение `PORT=8090`, и приложение смогло запуститься на новом порту.

### 8. Переключение трафика между ревизиями

После создания второй ревизии была проверена возможность управления трафиком между ревизиями.

В разделе **Revisions** были доступны две ревизии:

`evoroshnina-cloudrun-lab2-00001-8gb`

`evoroshnina-cloudrun-lab2-00002-bmr`

Изначально 100% трафика было направлено на последнюю ревизию:

`evoroshnina-cloudrun-lab2-00002-bmr`

Затем с помощью функции **Manage traffic** трафик был переключен на одну выбранную ревизию.

Был выбран вариант:

`Send all traffic to one revision`

После этого трафик был направлен на старую ревизию:

`evoroshnina-cloudrun-lab2-00001-8gb`

После переключения трафика сервис продолжил работать корректно.

Таким образом, была проверена возможность переключения трафика между разными ревизиями Cloud Run.

### 9. Удаление созданного сервиса

После завершения лабораторной работы созданный Cloud Run сервис был удален.

Для удаления была использована команда в Cloud Shell:

`gcloud run services delete evoroshnina-cloudrun-lab2 --region=europe-west1 --project=cloud-platforms-as-the-basis`

После подтверждения удаления сервис был удален из проекта.


## Вывод

В ходе лабораторной работы был изучен сервис Google Cloud Run.

Был создан Cloud Run service `evoroshnina-cloudrun-lab2` на основе стандартного контейнерного образа Hello. Для сервиса были заданы минимальные параметры ресурсов: `CPU 1`, `Memory 512 MiB`, минимальное количество инстансов `0` и максимальное количество инстансов `1`.

После развертывания сервис был протестирован по публичному URL. Страница `It's running!` подтвердила, что контейнер был успешно запущен и доступен из интернета.

Также были изучены разделы **Logs** и **Metrics**. В логах были отображены события создания сервиса, запуска контейнера и успешные HTTP-запросы со статусом `200`. В метриках были отображены графики количества запросов, задержек, состояния контейнера и параметров масштабирования.

После изменения порта контейнера с `8080` на `8090` была создана новая ревизия сервиса. Сервис продолжил работать, так как стандартный Hello-контейнер использует переменную окружения `PORT`, которую Cloud Run передает контейнеру автоматически.

Также было протестировано переключение трафика между ревизиями. Это позволило убедиться, что Cloud Run поддерживает управление версиями приложения и позволяет направлять трафик на выбранные ревизии.

После завершения работы созданный сервис был удален.
